CHƯƠNG 3. PHÂN TÍCH VÀ THIẾT KẾ HỆ THỐNG
3.1. Giới thiệu chương
Chương này trình bày quá trình phân tích và thiết kế hệ thống truyền tin bảo mật giữa các thiết bị IoT. Hệ thống được xây dựng theo mô hình gồm thiết bị cảm biến Xi, Gateway Y, Flask Server, cơ sở dữ liệu SQLite và giao diện giám sát Streamlit Dashboard.
Nội dung chính của chương tập trung vào việc xác định bài toán cần giải quyết, phân tích yêu cầu hệ thống, thiết kế kiến trúc tổng thể, thiết kế luồng truyền dữ liệu, thiết kế cơ chế mã hóa AES-128-CBC, thiết kế cơ chế chống tấn công phát lại bằng sequence number, thiết kế cơ sở dữ liệu và thiết kế giao diện giám sát.
Khác với chương thực nghiệm, Chương 3 không đi sâu vào từng đoạn mã nguồn cụ thể mà tập trung mô tả hệ thống được tổ chức như thế nào, các thành phần tương tác với nhau ra sao và dữ liệu được xử lý qua từng bước như thế nào. Đây là cơ sở để triển khai chương trình và kiểm thử hệ thống trong Chương 4.
3.2. Phân tích bài toán
Trong các hệ thống Internet of Things, dữ liệu thường được thu thập từ nhiều thiết bị cảm biến đặt phân tán tại nhiều vị trí khác nhau. Các thiết bị này có thể được triển khai trong môi trường ngoài trời, khu vực rộng, vùng xa hoặc nơi có hạ tầng Internet không ổn định. Do đó, việc kết nối trực tiếp từng thiết bị cảm biến với Server thông qua Wi-Fi hoặc mạng di động không phải lúc nào cũng phù hợp.
Một giải pháp thường được sử dụng là tổ chức hệ thống theo mô hình nhiều tầng. Trong đó, các node cảm biến chỉ truyền dữ liệu đến một Gateway gần nhất. Gateway có nhiệm vụ thu thập dữ liệu từ các node và chuyển tiếp lên Server trung tâm. Cách tổ chức này giúp giảm yêu cầu mỗi node phải kết nối Internet trực tiếp, đồng thời phù hợp với các hệ thống sử dụng công nghệ truyền thông không dây tầm xa như LoRa.
Trong đề tài này, hệ thống được thiết kế theo mô hình:
Node Xi → Gateway Y → Flask Server → SQLite Database → Streamlit Dashboard
Node Xi là thiết bị cảm biến có nhiệm vụ tạo hoặc thu thập dữ liệu. Dữ liệu này có thể bao gồm nhiệt độ, độ ẩm, áp suất, nồng độ khí, nhịp tim, SpO2 và vị trí GPS. Sau khi tạo dữ liệu, node Xi đóng gói dữ liệu dưới dạng JSON, bổ sung sequence number, mã hóa bằng AES-128-CBC và gửi đến Gateway Y.
Gateway Y đóng vai trò trung gian. Gateway tiếp nhận dữ liệu mã hóa từ node Xi, sau đó chuyển tiếp dữ liệu đến Flask Server thông qua Wi-Fi và giao thức HTTP. Gateway không thực hiện giải mã dữ liệu, do đó nội dung dữ liệu gốc vẫn được bảo vệ trong quá trình đi qua thiết bị trung gian.
Flask Server là thành phần xử lý trung tâm. Server nhận payload từ Gateway, kiểm tra định dạng, tách IV, giải mã dữ liệu, loại bỏ padding, chuyển dữ liệu về JSON, kiểm tra định danh thiết bị, kiểm tra sequence number và lưu dữ liệu hợp lệ vào SQLite. Nếu dữ liệu không hợp lệ hoặc có dấu hiệu bị phát lại, Server từ chối xử lý và không ghi dữ liệu vào bảng telemetry.
Streamlit Dashboard đọc dữ liệu từ SQLite để hiển thị cho người dùng. Dashboard giúp quan sát dữ liệu cảm biến, vị trí thiết bị, lịch sử truyền dữ liệu và kết quả benchmark của Server.
Bài toán của hệ thống không chỉ là truyền dữ liệu từ node đến Server mà còn phải đảm bảo một số yêu cầu quan trọng về bảo mật và tính đúng đắn của dữ liệu. Vì dữ liệu được truyền qua môi trường không dây nên có thể bị nghe lén hoặc thu thập bởi đối tượng bên ngoài. Nếu dữ liệu được truyền ở dạng rõ, kẻ tấn công có thể đọc được nội dung cảm biến, vị trí thiết bị hoặc các trạng thái quan trọng khác.
Ngoài ra, hệ thống còn phải đối mặt với nguy cơ tấn công phát lại. Trong kiểu tấn công này, một gói tin hợp lệ đã được gửi trước đó bị thu lại và gửi lại vào thời điểm khác. Nếu Server chỉ kiểm tra xem dữ liệu có giải mã được hay không, Server có thể chấp nhận gói tin cũ như một gói tin mới. Điều này làm sai lệch dữ liệu giám sát và ảnh hưởng đến độ tin cậy của hệ thống.
Vì vậy, bài toán đặt ra gồm ba nhóm yêu cầu chính.
Thứ nhất là yêu cầu truyền dữ liệu. Node Xi phải tạo dữ liệu, đóng gói dữ liệu, gửi đến Gateway và Gateway phải chuyển tiếp dữ liệu lên Server.
Thứ hai là yêu cầu bảo mật. Dữ liệu phải được mã hóa trước khi truyền để người ngoài không thể đọc trực tiếp nội dung plaintext. Trong hệ thống này, AES-128-CBC được sử dụng để mã hóa dữ liệu.
Thứ ba là yêu cầu chống phát lại. Mỗi gói dữ liệu cần có sequence number. Server lưu sequence number cuối cùng của từng thiết bị và chỉ chấp nhận gói tin có sequence number mới hơn. Cơ chế này giúp phát hiện các gói tin cũ bị gửi lại.
Từ các phân tích trên, hệ thống cần giải quyết các nhiệm vụ sau:
Mô phỏng hoặc thu thập dữ liệu cảm biến từ node Xi.
Thiết kế quá trình nhận biết Gateway thông qua Beacon và ACK.
Đóng gói dữ liệu cảm biến dưới dạng JSON.
Bổ sung sequence number vào dữ liệu.
Mã hóa dữ liệu trên node Xi bằng AES-128-CBC.
Chuyển dữ liệu mã hóa thành payload dạng hex.
Gửi payload từ node Xi đến Gateway Y.
Chuyển tiếp payload từ Gateway Y đến Flask Server.
Giải mã dữ liệu tại Server.
Kiểm tra sequence number để phát hiện replay.
Lưu dữ liệu hợp lệ vào SQLite.
Hiển thị dữ liệu trên Dashboard.
Ghi nhận benchmark để đánh giá hiệu năng xử lý.
3.3. Mục tiêu thiết kế hệ thống
Mục tiêu thiết kế của hệ thống là xây dựng một mô hình truyền tin IoT có khả năng thu thập, mã hóa, truyền, giải mã, kiểm tra, lưu trữ và hiển thị dữ liệu.
Các mục tiêu cụ thể gồm:
3.3.1. Thiết kế kiến trúc tổng thể
Hệ thống phải được chia thành các thành phần rõ ràng gồm node Xi, Gateway Y, Server, cơ sở dữ liệu và Dashboard. Mỗi thành phần có nhiệm vụ riêng và giao tiếp với nhau thông qua các định dạng dữ liệu được xác định trước.
3.3.2. Thiết kế giao tiếp giữa node Xi và Gateway Y
Node Xi cần có cơ chế thông báo sự có mặt của mình cho Gateway thông qua Beacon. Gateway sau khi nhận Beacon cần phản hồi ACK để xác nhận có thể tiếp nhận dữ liệu từ node.
3.3.3. Thiết kế cấu trúc dữ liệu cảm biến
Dữ liệu cảm biến cần được tổ chức rõ ràng, dễ xử lý và dễ mở rộng. JSON được lựa chọn để đóng gói dữ liệu trước khi mã hóa.
3.3.4. Thiết kế cơ chế mã hóa
Dữ liệu trước khi truyền phải được mã hóa bằng AES-128-CBC. Quá trình mã hóa cần có khóa 16 byte, IV 16 byte và cơ chế padding để bảo đảm dữ liệu phù hợp với kích thước khối của AES.
3.3.5. Thiết kế payload truyền dữ liệu
Dữ liệu sau mã hóa gồm IV và ciphertext. Hai thành phần này được ghép lại, sau đó chuyển thành chuỗi hex để đóng gói trong JSON và gửi qua HTTP.
3.3.6. Thiết kế Server xử lý dữ liệu
Server cần tiếp nhận dữ liệu từ Gateway, kiểm tra payload, giải mã, kiểm tra JSON, kiểm tra sequence number, lưu dữ liệu và trả kết quả xử lý.
3.3.7. Thiết kế cơ chế chống replay
Mỗi node có một sequence number tăng dần. Server lưu sequence number cuối cùng đã được chấp nhận của từng node. Nếu nhận gói tin có sequence number cũ hơn hoặc bằng giá trị đã lưu, Server từ chối gói tin.
3.3.8. Thiết kế cơ sở dữ liệu
Cơ sở dữ liệu cần lưu được thông tin thiết bị, dữ liệu telemetry, dữ liệu GPS, sequence number và benchmark xử lý.
3.3.9. Thiết kế Dashboard
Dashboard cần hiển thị dữ liệu cảm biến, vị trí thiết bị, lịch sử dữ liệu và thông tin benchmark để phục vụ quan sát và đánh giá hệ thống.

3.5.2. Vai trò của từng tầng
a. Tầng thiết bị cảm biến
Tầng thiết bị cảm biến gồm các node Xi. Đây là nơi dữ liệu được tạo ra. Node Xi có thể được triển khai trên ESP32 hoặc board TTGO T-Beam. Trong hệ thống thực tế, node Xi có thể đọc dữ liệu từ cảm biến và GPS. Trong hệ thống mô phỏng, dữ liệu được tạo ngẫu nhiên.
Tầng này cũng là nơi thực hiện mã hóa dữ liệu. Việc mã hóa được thực hiện ngay tại node giúp dữ liệu được bảo vệ trước khi truyền ra môi trường không dây.
b. Tầng Gateway
Gateway Y là cầu nối giữa mạng cảm biến và Server trung tâm. Gateway nhận Beacon, gửi ACK và chuyển tiếp payload mã hóa. Gateway không thực hiện giải mã nên không cần lưu khóa AES. Điều này giúp thiết kế hệ thống bảo mật hơn vì nếu Gateway bị truy cập trái phép, dữ liệu plaintext vẫn không bị lộ trực tiếp.
c. Tầng Server
Flask Server tiếp nhận dữ liệu từ Gateway. Đây là nơi duy nhất có khả năng giải mã payload. Server thực hiện kiểm tra dữ liệu, phát hiện replay, lưu database và ghi benchmark.
d. Tầng lưu trữ
SQLite Database lưu dữ liệu sau khi Server xử lý. Cơ sở dữ liệu gồm thông tin thiết bị, dữ liệu telemetry và dữ liệu benchmark. Đây là nguồn dữ liệu cho Dashboard.
e. Tầng hiển thị
Streamlit Dashboard hiển thị dữ liệu đã lưu. Người dùng có thể xem dữ liệu cảm biến, lịch sử dữ liệu, biểu đồ, vị trí thiết bị và kết quả hiệu năng.
3.5.3. Đặc điểm của kiến trúc
Kiến trúc này có một số đặc điểm chính:
Node Xi là nơi tạo và mã hóa dữ liệu.
Gateway Y chỉ chuyển tiếp dữ liệu, không giải mã.
Server là nơi giải mã và xác thực dữ liệu.
SQLite lưu toàn bộ dữ liệu đã xử lý.
Dashboard không xử lý mã hóa mà chỉ đọc dữ liệu đã giải mã từ database.
Các thành phần được tách biệt nên dễ nâng cấp và kiểm thử.
3.6. Thiết kế node Xi
3.6.1. Vai trò của node Xi
Node Xi là thiết bị cảm biến trong hệ thống. Node có nhiệm vụ tạo dữ liệu, bảo vệ dữ liệu và gửi dữ liệu đến Gateway. Trong mô hình thực tế, node Xi có thể sử dụng ESP32 kết hợp với module LoRa, GPS và các cảm biến. Trong mô hình thử nghiệm, node Xi được mô phỏng để kiểm tra luồng xử lý.
Node Xi là thành phần quan trọng nhất ở phía thiết bị vì dữ liệu phải được xử lý ngay từ nguồn. Nếu dữ liệu không được mã hóa tại node, dữ liệu có thể bị lộ khi truyền qua môi trường không dây. Vì vậy, node Xi cần thực hiện đầy đủ quá trình tạo JSON, thêm sequence number, mã hóa AES-128-CBC và tạo payload.

3.8. Thiết kế quy trình mã hóa AES-128-CBC
3.8.1. Lý do sử dụng AES-128-CBC
Dữ liệu truyền qua môi trường không dây có nguy cơ bị nghe lén. Nếu truyền JSON trực tiếp, người ngoài có thể đọc được nội dung như nhiệt độ, độ ẩm, vị trí GPS hoặc sequence number. Vì vậy, dữ liệu cần được mã hóa trước khi gửi.
AES là thuật toán mã hóa đối xứng được sử dụng rộng rãi. AES-128 sử dụng khóa 128 bit, tương ứng 16 byte. Chế độ CBC sử dụng IV để làm cho quá trình mã hóa an toàn hơn so với việc mã hóa từng khối độc lập.
Trong hệ thống, AES-128-CBC được lựa chọn vì:
Có độ bảo mật phù hợp với mô hình thử nghiệm.
Có thể triển khai trên ESP32.
Phù hợp với dữ liệu dạng JSON có kích thước nhỏ.
Server có thể giải mã dễ dàng bằng cùng khóa bí mật.
Có thể kết hợp với IV để tránh cùng plaintext tạo cùng ciphertext.
3.8.2. Dữ liệu đầu vào của quá trình mã hóa
Dữ liệu đầu vào là chuỗi JSON chứa dữ liệu cảm biến, GPS, Gateway và sequence number.
Ví dụ:
{
  "id": "Xi_01",
  "t": 28.5,
  "h": 65.2,
  "gw": "Y_01",
  "seq": 1001
}
Chuỗi JSON này được chuyển thành mảng byte trước khi mã hóa.
3.8.3. Khóa AES
AES-128 sử dụng khóa 16 byte.
Ở mức thiết kế:
Key = 16 byte bí mật được cấu hình trước ở node Xi và Server
Khóa chỉ tồn tại ở node Xi và Server. Gateway không cần khóa vì Gateway không giải mã dữ liệu.
Trong báo cáo, không nên ghi khóa thật nếu hệ thống có sử dụng khóa cụ thể trong mã nguồn. Có thể mô tả khóa ở mức nguyên lý để tránh lộ thông tin bảo mật.
3.8.4. Padding dữ liệu
AES-CBC xử lý dữ liệu theo từng khối 16 byte. Tuy nhiên, chuỗi JSON có thể có độ dài bất kỳ. Vì vậy, trước khi mã hóa cần thêm padding để độ dài plaintext trở thành bội số của 16.
Công thức:
padding_length = 16 - (plaintext_length mod 16)
Ví dụ:
plaintext_length = 45 byte
45 mod 16 = 13
padding_length = 16 - 13 = 3
Khi đó hệ thống thêm 3 byte padding.
Sau khi giải mã, Server dựa vào giá trị padding để loại bỏ phần dữ liệu thêm vào và khôi phục lại JSON ban đầu.
3.8.5. IV trong AES-CBC
IV là viết tắt của Initialization Vector. IV có độ dài 16 byte, bằng kích thước khối AES.
Trong hệ thống:
Mỗi lần mã hóa tạo một IV mới.
IV không cần giữ bí mật.
IV được ghép vào đầu ciphertext.
Server tách 16 byte đầu tiên để lấy IV.
Phần còn lại là ciphertext.
Việc sử dụng IV giúp cùng một dữ liệu JSON khi mã hóa ở hai lần khác nhau có thể tạo ra ciphertext khác nhau.
3.8.6. Cấu trúc payload sau mã hóa
Sau khi mã hóa, payload có cấu trúc:
Payload byte = IV || Ciphertext
Trong đó:
Sau đó payload byte được chuyển thành chuỗi hex:
Payload hex = HEX(IV || Ciphertext)
Ví dụ minh họa:
IV:         16 byte đầu
Ciphertext: các byte còn lại
Payload:   IV + Ciphertext

3.10. Thiết kế cơ chế chống tấn công phát lại
3.10.1. Tấn công phát lại trong hệ thống IoT
Tấn công phát lại xảy ra khi kẻ tấn công thu thập một gói tin hợp lệ và gửi lại gói tin đó đến Server. Vì gói tin ban đầu là hợp lệ nên nếu Server không kiểm tra tính mới của dữ liệu, Server có thể chấp nhận gói tin cũ.
Ví dụ, node Xi gửi dữ liệu:
seq = 1001
temperature = 28.5
humidity = 65.2
Server nhận và lưu dữ liệu này. Nếu kẻ tấn công gửi lại đúng payload đó, Server vẫn có thể giải mã thành công. Tuy nhiên, dữ liệu này không còn là dữ liệu mới. Nếu Server tiếp tục lưu, hệ thống sẽ ghi nhận trùng lặp hoặc sai trạng thái.
3.10.2. Nguyên tắc sử dụng sequence number
Để chống replay, mỗi gói dữ liệu có một sequence number. Sequence number tăng dần theo thời gian gửi.
Ví dụ:
Gói thứ nhất: seq = 1001
Gói thứ hai:  seq = 1002
Gói thứ ba:   seq = 1003
Server lưu sequence number cuối cùng đã chấp nhận của từng node.
Ví dụ:
Xi_01 → last_seq = 1002
Xi_02 → last_seq = 501
Khi nhận gói mới từ Xi_01, Server chỉ chấp nhận nếu:
seq_received > last_seq
3.10.3. Thuật toán kiểm tra sequence number
Thuật toán kiểm tra:
Input: device_id, seq_received

Bước 1: Lấy last_seq của device_id từ database

Bước 2:
Nếu chưa có device_id trong database:
    Tạo device mới
    Chấp nhận seq_received

Bước 3:
Nếu device_id đã tồn tại:
    So sánh seq_received với last_seq

Bước 4:
Nếu seq_received > last_seq:
    Chấp nhận dữ liệu
    Cập nhật last_seq = seq_received

Bước 5:
Nếu seq_received <= last_seq:
    Từ chối dữ liệu
    Ghi nhận trạng thái replay
3.10.4. Ví dụ kiểm tra sequence
Trường hợp hợp lệ
Database đang lưu:
Xi_01: last_seq = 1001
Server nhận:
Xi_01: seq_received = 1002
Vì:
1002 > 1001
Server chấp nhận dữ liệu và cập nhật:
Xi_01: last_seq = 1002
Trường hợp replay
Database đang lưu:
Xi_01: last_seq = 1002
Server nhận:
Xi_01: seq_received = 1002
Vì:
1002 <= 1002
Server từ chối gói tin.
Trường hợp sequence thấp hơn
Database đang lưu:
Xi_01: last_seq = 1002
Server nhận:
Xi_01: seq_received = 1000
Vì:
1000 <= 1002
Server từ chối gói tin.
3.10.5. Ưu điểm của sequence number
Cơ chế sequence number có các ưu điểm:
Dễ triển khai.
Phù hợp với thiết bị IoT tài nguyên hạn chế.
Không làm tăng kích thước dữ liệu quá nhiều.
Server có thể kiểm tra nhanh.
Có thể quản lý riêng cho từng node.
Phát hiện được gói tin cũ bị gửi lại.
3.10.6. Hạn chế của sequence number
Cơ chế này vẫn có một số hạn chế:
Nếu node reset và sequence number quay về 0, Server có thể từ chối dữ liệu mới.
Nếu database bị mất, Server mất thông tin last_seq.
Sequence number không xác thực nguồn gửi.
Sequence number không kiểm tra toàn vẹn dữ liệu.
Cần kết hợp thêm HMAC hoặc mã hóa xác thực để tăng bảo mật.
Hướng khắc phục là lưu sequence number vào bộ nhớ không mất dữ liệu của ESP32, đồng bộ lại sequence number khi node khởi động hoặc sử dụng thêm timestamp, nonce và HMAC.

3.13. Thiết kế luồng xử lý toàn hệ thống
3.13.1. Luồng dữ liệu hợp lệ
Node Xi          Gateway Y          Flask Server          SQLite
   │                 │                    │                  │
   │── Beacon ──────>│                    │                  │
   │<── ACK ─────────│                    │                  │
   │                 │                    │                  │
   │ Tạo dữ liệu     │                    │                  │
   │ Tạo JSON        │                    │                  │
   │ Thêm seq        │                    │                  │
   │ Mã hóa AES      │                    │                  │
   │                 │                    │                  │
   │── Payload ─────>│                    │                  │
   │                 │── HTTP POST ──────>│                  │
   │                 │                    │ Tách IV          │
   │                 │                    │ Giải mã AES      │
   │                 │                    │ Kiểm tra seq     │
   │                 │                    │── INSERT ───────>│
   │                 │                    │── UPDATE seq ───>│
   │                 │                    │<── OK ───────────│
   │                 │<── HTTP 200 ───────│                  │
   │<── Trạng thái ──│                    │                  │
Luồng này mô tả trường hợp dữ liệu hợp lệ, sequence number mới và Server lưu thành công vào database.
3.13.2. Luồng xử lý replay
Gateway Y          Flask Server          SQLite
   │                    │                  │
   │── HTTP POST ──────>│                  │
   │                    │ Tách IV          │
   │                    │ Giải mã AES      │
   │                    │ Lấy device_id    │
   │                    │ Lấy seq          │
   │                    │── Đọc last_seq ─>│
   │                    │<── last_seq ─────│
   │                    │ So sánh seq      │
   │                    │ Phát hiện replay │
   │<── HTTP Error ─────│                  │
Ở trường hợp này, payload vẫn có thể giải mã được nhưng sequence number không hợp lệ. Server không ghi dữ liệu vào bảng telemetry.
3.13.3. Luồng xử lý lỗi giải mã
Gateway Y          Flask Server
   │                    │
   │── HTTP POST ──────>│
   │                    │ Kiểm tra payload
   │                    │ Tách IV
   │                    │ Giải mã AES
   │                    │ Giải mã thất bại
   │<── Error response ─│
Trường hợp này xảy ra khi payload sai, ciphertext bị thay đổi, khóa không đúng hoặc padding không hợp lệ.
